## ⚠️ Method & Scope Disclaimer (read first)

This assessment combined a **white-box source-code audit** (repository at `main`/HEAD) with **live testing of the production site** in two passes: (1) unauthenticated public probing, and (2) — after test credentials were provided — an **authenticated pass driven via scripted HTTP** (NextAuth `CSRF → login → session-cookie` via `curl`; no browser). See **§0 Live Verification Results** below for what pass (2) confirmed on the running build.

**What this means for how you read this report:**

- **Authenticated testing was done via scripted HTTP, not a browser.** I logged in as all five roles and drove protected routes/APIs with the real session cookie — so login, role-landing, RBAC URL-tampering, and several findings are **live-verified** (marked ✅ Live below). With no browser I still could **not** capture screenshots, measure mobile/responsive layout, read the browser console, or exercise client-side JS flows (e.g. Stripe Checkout) — those remain **Not Tested**.
- **Destructive/privileged writes were deliberately withheld on production.** Findings provable only by *mutating* real state (P0 message-send, P1-A role self-promotion, P2-D certificate fraud) were confirmed in source and **not executed** against the live site, to avoid altering production data. The exact requests can be run on staging.
- **Findings are code-verified** (every P0/P1/P2 was read first-hand at the cited `file:line`), but should be **re-confirmed against the running production build** before/while fixing. The audit is of source at HEAD; I could not prove the deployed image is built from the exact same commit. (The live `/en/courses → /courses` redirect is explained by the repo's `localePrefix: "as-needed"` config — [src/i18n/navigation.ts:8](src/i18n/navigation.ts#L8) — i.e. consistent with HEAD, but not a guarantee.)
- **Live facts I *did* confirm:** the site is up, Cloudflare-fronted, `GET /api/health` returns `{"status":"ok","db":"ok","latencyMs":1}`, and the public catalog renders 25+ real, age-banded courses via SSR.
- Items that genuinely require a running session (real login per role, session-expiry behavior, mobile/responsive, console errors, live Stripe flows) are marked **Not Tested** — they are not failures, they are untested.

**Severity scale:** **P0** blocker (prevents launch / exposes private or safety-sensitive data) · **P1** high (major broken journey or privilege issue) · **P2** medium (real issue, workaround exists) · **P3** low (polish/UX/content).

**Production reframe:** Schulab **soft-launched ~2026-06-25** and is live today. The issues below are therefore **production hotfixes**, not pre-launch blockers — which *raises* urgency for the safety and privilege items.

## 1. Executive Summary

Schulab is an **architecturally mature, genuinely polished, feature-complete** LMS. The engineering quality is high: clean role-gating via server-component layouts, a safe-by-design admin "view-as" preview, correct subscription/consent gating primitives, comprehensive empty-states and error boundaries, and **excellent internationalization** (1044/1044 message keys across en/ar/de with real Arabic/German translations and correct RTL). The public catalog is real and thoughtfully structured (STEM, ages 3–18, six clean age bands, free/subscription mix). Cross-user data isolation is **solid** — I found **no P0 IDOR / data-breach**: parents can only see linked children, tutors only assigned students, message *reads* and notifications are owner-scoped.

**What works well**
- RBAC model is sound; admin mutations, cron jobs and webhooks are consistently guarded (secrets use constant-time comparison, fail-closed in prod).
- No cross-user private-data exposure via the obvious IDOR vectors (parent→child IDOR **live-verified**: an unrelated real student returns `404`).
- i18n, empty-states, error boundaries, and catalog content are launch-grade.
- Payments/consent primitives are correct (PAST_DUE excluded from entitlement; missing-DOB treated as "undeclared", not "adult"; subscription access correctly revokes on cancellation).

**Biggest blockers (all live in production now)**
1. **[P0 — Safeguarding] Unrestricted direct messaging on a minors' platform.** Any authenticated user can DM *any* other user by id — including adult→minor — with only an "exists" check on the recipient. (The public leaderboard and reviews I audited are **server-rendered and do not expose user ids** to the browser, so trivial mass-harvest isn't demonstrated — but *any* known student id is a valid, unguarded target.) This is the single most urgent item.
2. **[P1 — Privilege escalation] A plain ADMIN can promote themselves to SUPER_ADMIN** via `updateUserRole` (wrong guard), collapsing the ADMIN/SUPER_ADMIN boundary the app otherwise enforces.
3. **[P1 — Broken login] `B2B_PARTNER` accounts hit an infinite redirect loop; `ORG_ADMIN` accounts are stranded on `/onboarding`** — both roles are seeded, so these are active, not theoretical.

**Readiness verdict:** **Internal-beta grade today; the P0 messaging + P1 privilege items are production hotfixes.** The core product is sound and already serving. But because it is a live platform for children, the open-messaging safeguarding gap should be treated as urgent (days, not sprints — the fixes are small and well-scoped). See §10.

## 5. Role-Based Findings

### 5.1 Admin
- **[P1-A] Self-escalation to SUPER_ADMIN.** *What:* `updateUserRole` uses `requireAdmin`. *Why it matters:* the app deliberately reserves settings/permission mutations for SUPER_ADMIN — this action lets any ADMIN bypass that wall. It **is** audit-logged (`user.update_role`), so it's traceable, but not prevented. *Assumption:* this is P1 because ADMIN is clearly a lesser tier than SUPER_ADMIN here (3 ADMIN vs 5 SUPER_ADMIN seeded; SUPER_ADMIN-only guards exist). If in practice all admins are equally trusted, treat as P2. *Fix:* `requireSuperAdmin` + self/target guards.
- **[P3-F] `toggleUserActive` lets an ADMIN deactivate a SUPER_ADMIN** ([admin.actions.ts:45](src/actions/admin.actions.ts#L45)) — authority inversion; low impact but same root cause as P1-A.
- **[P2-E] No content-health/empty-content visibility** (see table). This directly answers the brief's "can admin identify missing content / empty lessons / unpublished courses?" → **not in the UI today.**
- **Positives:** every other admin mutation across 16 action files is correctly `requireAdmin`/`requireSuperAdmin`-guarded; cron routes use constant-time Bearer checks and fail closed in prod; the GitHub curriculum webhook verifies HMAC-SHA256; the curriculum media route is authenticated, enrollment-gated, and path-traversal-safe (rejects `..`, and the sink is the GitHub Contents API, not the local FS).

### 5.2 Tutor
- **Scope is correct:** the tutor "students" list derives solely from `getBookingsForTutor(tutor.id)` (scoped by `tutorId`), and there is **no `tutor/students/[id]` route** to bypass by id. No cross-tutor leakage found — ✅.
- **Content authoring is admin-only.** Tutors cannot create/edit catalog courses/lessons (those actions are `requireAdmin`). Tutors interact with content via live sessions/bookings. *Product gap to confirm:* the brief expects "ability to add, edit, or review content if allowed" — today tutors have **no catalog-authoring surface**; verify this matches intended product scope.
- **Runtime Not Tested:** tutor dashboard, availability grid, booking management, and live-classroom (LiveKit) all exist in code but need a logged-in tutor to validate behavior.

### 5.3 Student
- **[P2-C] Quiz answer-key leak** and **[P2-D] completion fraud / prereq bypass** — see table. Both are integrity issues (cheating, unearned certificates), not privacy breaches.
- **[P3-D] `/student/*` is authenticated but not role-gated — confirmed live.** ✅ Live: `/student` returned `200` for admin, tutor, *and* parent sessions (vs `307→/dashboard` for `/tutor` and `/parent`). Unlike `/tutor` and `/parent` (which have role-checking layouts), the `student` segment has **no `layout.tsx`**, and `student/page.tsx` checks only `if (!session) redirect("/login")` ([student/page.tsx:41-42](src/app/[locale]/(dashboard)/student/page.tsx#L41-L42)). The tutor's `/student` view rendered tutor-onboarding strings (no student data) — so it shows *the viewer's own* view; no other user's data leaks. Consistency/UX issue, not a security one.
- **[P3] Lesson SSR page checks enrollment *existence*, not `ACTIVE` status/plan**, diverging from the stricter `checkLessonAccess` used by the APIs — a COMPLETED/withdrawn enrollment can still render the SSR page. Low impact; align the two gates.
- **Consistency between the two student accounts (aisha/liam):** both are plain STUDENTs with identical code paths — behavior will be consistent by construction (age-appropriateness depends on their `dateOfBirth`/consent, which is data, not code). **Runtime Not Tested.**
- **Positives:** enrollment is self-only (no attacker-supplied `userId`); free vs paid gating is enforced at enroll *and* access; certificates issue only on a genuine 100% transition; quiz *submission* is enrollment-gated.

### 5.4 Parent
- **No cross-child data leak (the headline privacy risk) — verified safe, live.** ✅ Live test: as Sara, her own child returned `200`, unrelated real student **Liam returned `404`**, and a bogus id returned `404`. The child-detail page resolves children via `getChildren(session.user.id)` (scoped to `ParentChild`) and gates on `children.find(...) → notFound()` **before** loading any child data ([parent/children/[childId]/page.tsx:43-45](src/app/[locale]/(dashboard)/parent/children/[childId]/page.tsx#L43-L45)). Every parent mutation (consent grant/withdraw, enroll, remove) re-checks the `ParentChild` link server-side. A parent cannot view or act on an unlinked child by changing the URL id. ✅ (matches `parent.actions.test.ts`.)
- **Latent fragility (P3):** `getUserById` / `getUserEnrollments` / `getStudentStats` are internally *unscoped* — safe only because current call sites gate the id (`getUserById` is dead code today). A future caller passing a raw user id without a link check would open an IDOR. Recommend scoping these helpers defensively.
- **Runtime Not Tested:** the parent billing/purchase-for-child flow and notification/digest emails need a live session.

## 6. Content & Curriculum Gaps

- **Catalog is genuinely populated and age-appropriate (positive).** Live SSR renders 25+ real courses across six clean bands — *Wonder Lab: Science for Tiny Explorers* (3-5) … *Python for Real-World Projects*, *AI Project Builder*, *Tech Innovator Portfolio* (14-18). Age bands are a **fixed, drift-proof enum** (`AGES_3_5 … AGES_17_18`, single-sourced in [constants.ts](src/lib/constants.ts)), assigned per course in the admin editor. No age-mismatch, duplicate, or lorem/placeholder content detected in seed or sync source. (Note: `&amp;` in some titles is correct HTML escaping, not a bug.)
- **Structure vs content separation (as the brief asks):** confirmed. Course/module/lesson **structure, titles and age-group tags** are managed on the platform (admin editor + a fixed enum), while **curriculum lesson content** is synced separately from the external STEM-Curricula GitHub repo (webhook + cron). Seed creates 25 course *shells* (only 1 demo module/lesson); real content is populated by the sync, which has clearly run in production.
- **[P2-E] No admin visibility of content gaps.** There is no admin surface to flag **empty lessons, missing media, or unpublished courses**; the one health service that exists (`listWeakLessons`) is (a) not wired to any UI and (b) scores engagement, not emptiness. **This is the most important content-side gap.**
- **Media integrity — Not Tested (needs runtime).** Lesson media (images/PDFs/videos) resolve through the external curriculum repo via the media proxy route. I could not verify individual assets render (no logged-in session / no access to the external repo from here). **Recommend a runtime spot-check** across a sample of lessons per age band for broken images/links/PDFs.
- **Recommendation gaps (from code):** the student dashboard surfaces "recommended next" (pathway/bundle/next course) and **event/competition recommendations** already (`EventRecommendations`, `getEligibleEventsForUser`). Post-completion event/competition prompts exist in code but their *quality/coverage* (does every completed course map to a relevant next step / competition?) is **data-dependent and Not Tested** — worth a content review pass.

## 10. Final Launch Recommendation

**Ready only for internal beta / a limited, monitored soft-launch as-is — with the P0 messaging and P1 privilege items treated as immediate production hotfixes.**

Schulab is **not** "not ready" — it is a well-built, already-live platform with sound architecture, real localized content, and no gross data-exposure IDORs. For a **small, supervised audience** it is usable today. However, because it is a **live service for children**, I cannot recommend *broad* public promotion until:

1. **[P0] Direct messaging is relationship-restricted.** Today any authenticated user can DM any user id with only an existence check — on a minors' platform that is a safeguarding requirement, not a nice-to-have, and it is live in production **now**. (The public leaderboard/reviews I audited are server-rendered and do *not* expose user ids, so this is confirmed *targeted* abuse rather than trivially mass-scale — still a launch-grade safeguarding gap.)
2. **[P1] The admin→super-admin self-escalation is closed** and **B2B_PARTNER/ORG_ADMIN login is fixed** (those partner/school accounts currently can't get in — a problem if B2B onboarding is part of the launch).

These fixes are **small and well-scoped** (guard swaps, a relationship check, a payload trim, a redirect-map fix) — realistically **days, not sprints**. The P2 cluster (consent-at-API, quiz/answer leak, progress integrity, content-health UI, session freshness) should follow immediately after as a fast-follow.

**Bottom line:** strong product, launch-grade polish, one urgent safeguarding gap and two high-priority auth fixes standing between "limited beta" and "confident public soft-launch." Prioritize items 1–3, then re-run a **live authenticated functional pass** (the black-box testing this source audit could not perform) to confirm the fixes and cover the runtime-only surfaces (mobile, console, payments, media).
